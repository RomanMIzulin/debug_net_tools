## ADR-001: Pure Go SQLite вместо CGO

**Дата:** 2025-02-26
**Статус:** Принято

**Контекст:** Нужно хранилище для сессий и фреймов. SQLite — единственный кандидат (встроенная БД, zero-config, один файл). Два драйвера: `mattn/go-sqlite3` (CGO) и `modernc.org/sqlite` (pure Go).

**Решение:** `modernc.org/sqlite`

**Аргументы:**
- Zero CGO = тривиальная кросс-компиляция
- Нет зависимости от C-компилятора в CI и у разработчиков
- Для concurrent reads (наш основной паттерн) modernc даже быстрее mattn
- Bulk write ~3.5x медленнее, но для debugging-прокси это не bottleneck

**Последствия:**
- (+) `go build` просто работает на любой платформе
- (+) Простой CI без C toolchain
- (-) Bulk inserts медленнее — если понадобится массовый импорт, нужен batch + transaction

## ADR-002: Proxy-подход вместо pcap/eBPF

**Дата:** 2025-02-26
**Статус:** Принято

**Контекст:** Два способа перехватывать WebSocket трафик: (A) MITM-прокси на application layer, (B) packet capture через pcap/eBPF на network layer.

**Решение:** Proxy (A)

**Аргументы:**
- Работает на application layer — не надо парсить сырые TCP-пакеты
- Не надо восстанавливать порядок пакетов и подгружать TLS-ключи
- Можно сосредоточиться на WebSocket-протоколе
- Go — язык для network-heavy сервисов, много библиотек

**Последствия:**
- (+) Значительно проще реализация
- (+) Полный доступ к расшифрованным данным
- (-) Клиент должен явно подключаться к прокси (не прозрачный перехват)
- (-) Вносит дополнительную latency (два hop вместо одного)

## ADR-003: Fan-out через каналы + file writer в ядре

**Дата:** 2025-02-26
**Статус:** Принято

**Контекст:** Ядро генерирует события (фреймы). Потребители: TUI, file writer, GUI. Нужен механизм распределения.

**Решение:**
- Fan-out через Go-каналы с drop-стратегией для TUI/stdout
- File writer — часть ядра, НЕ подписчик

**Аргументы:**
- TUI/stdout потеря кадра некритична → at-most-once с `select/default`
- File writer ДОЛЖЕН гарантировать запись → не подписчик, а часть pipeline
- Простая реализация: `Bus` с `Subscribe(ctx) <-chan Event`

**Последствия:**
- (+) TUI никогда не блокирует ядро
- (+) Персистентность гарантирована (file writer не drop-ает)
- (-) Дублирование: file writer и SQLite store оба получают данные напрямую из ядра
- Будущее: ring buffer если появится требование "подключиться к уже идущей сессии и увидеть последние N фреймов"

## ADR-004: squirrel как query builder

**Дата:** 2025-02-26
**Статус:** Принято

**Контекст:** Нужен инструмент для SQL-запросов к SQLite. Варианты: raw SQL, squirrel, sqlc, GORM.

**Решение:** `squirrel`

**Аргументы:**
- Не ORM — только строит SQL, без магии
- Хорош для динамических запросов (фильтры сессий и фреймов)
- Минимальная абстракция поверх `database/sql`
- GORM противоречит Go-философии (reflection, implicit behavior)
- sqlc отличный, но хуже для динамических WHERE-клаузов

**Последствия:**
- (+) Полный контроль над SQL
- (+) Динамические фильтры строятся естественно
- (-) Нет struct mapping — сканируем вручную или парим с sqlx
- (-) Нет compile-time validation запросов (sqlc это даёт)

## ADR-005: Go вместо Zig

**Дата:** 2025-02-26
**Статус:** Принято

**Контекст:** Выбор языка для прототипа WebSocket debugging proxy.

**Решение:** Go

**Аргументы:**
- Прототип + тестирование идеи — Go быстрее для этого
- Zig интересен, но работа напрямую с памятью и ОС замедлит
- Go исторически язык для network-heavy сервисов
- Богатая экосистема библиотек (gorilla/websocket, cobra, bubbletea)
- Goroutines + channels = естественный fit для bidirectional relay

**Последствия:**
- (+) Быстрый прототип
- (+) Learning goals по Go concurrency, net/http, CLI
- (-) Нет manual memory control (если производительность станет критичной)
